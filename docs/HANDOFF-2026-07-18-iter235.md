# PART 1 — THE STANDARD

The operator's instruction, stated directly: *hold the engineering standard of a top SpaceX team and the
scientific standard of a top Google DeepMind team.* That is not decoration. Here is what it means
operationally, and how it has actually been enforced in this repository.

## 1.1 The bar that matters most: attack your own numbers harder than anyone else will

This is the single most load-bearing habit in this programme, and it has paid out **four times in one
session**. Every one of these was a number that looked publishable and was my own artifact:

| Iteration | The number that looked good | What it actually was |
| --- | --- | --- |
| iter231 | `4/13` oracle recall | Two flags were broken exercises, not caught patches |
| iter232 | stage B rejecting 6 rows | **5 of 6 rejections were wrong** — the gate false-alarmed |
| iter234 | `3/10` on the value class | Two were tests that fail on *every* implementation |
| iter235 | `0/11` confirmed — a clean null | A slot-mapping bug made confirmation structurally impossible |

Three of those four moved the headline **down** when corrected. The fourth (iter235) moved it up, and was
only caught because *the judges were agreeing with each other while nothing confirmed* — a combination that
cannot happen if the comparison is sound.

**The discipline: when a result is surprisingly clean, surprisingly favourable, or surprisingly null, that is
the moment to distrust your instrument, not to write it up.** Ask what would have to be true of your code for
that number to be an artifact, then go check that specific thing.

## 1.2 Pre-register before data. Always. No exceptions.

Every experiment gets a `HYPOTHESIS.md` with numeric bars and named falsifiers, committed and git-timestamped
**before** any provider call. Git is the proof; that is why it is committed separately rather than written
alongside the result.

Two refinements this session added, both of which you should keep:

- **Pre-commit the correction.** iter234 stated in advance that if it caught a value-wrong patch, the paper's
  central claim would be narrowed from "no gold-free instrument" to "no gold-free instrument that reasons
  from the patch." It did, and the correction could not then be quietly avoided. Write the uncomfortable
  consequence into the pre-registration.
- **Amend before data, not after.** iter234 moved from one author to three. Deciding that *after* seeing the
  first author's result would have made the design conditioned on an outcome. The amendment is a separate
  commit with its own timestamp, and it says why.

The standing example of getting this wrong is iter200, whose strict two-judge rule was adopted *after*
outcome inspection (`3` loose versus `1` strict). That single choice is why iter200 is permanently labelled
exploratory rather than confirmatory. **One post-hoc rule costs a result its status forever.**

## 1.3 Nulls publish at full weight, and a null against your own expectation is the best kind

iter231 and iter232 are nulls. The pre-registered expectation was that a gold-free execution oracle would
beat static judging; it did not, and iter232 removed the excuse that the instrument was broken. Both are
published as prominently as any positive.

iter234's independence endpoint came back **underpowered** — I designed a measurement that could not answer
its own question. That is reported as unanswered rather than spun. Do the same.

## 1.4 Evidence or it did not happen

Every number must regenerate from committed artifacts. `python3 scripts/run_ci_closure.py --python
python3.11` runs **285** commands derived from `ci.yml`; every published figure has a `--check` guard that

Corollary that bit hard this session: **re-derive, do not trust the summary — and read the source, not the

## 1.5 World-contact beats self-consistency

A gate that passes only internal checks is satisfiable without touching reality, and a capable optimiser will
find that fixed point. Every gate needs at least one bar that can fail on the world.

Concretely, this session: the stage B probe passed every local check and died with `SyntaxError` in three
containers because it used `from __future__ import annotations` (3.7+) against a Python 3.6 image. A Gemini
test used a nested f-string with escaped quotes — legal on 3.12+, illegal before. **This host runs Python
3.14; the containers run 3.6 through 3.11. Local validation is systematically blind to this.** The shared
validity gate now hands source to `python3.11` and asks whether it parses — ground truth, after a hand-rolled
scanner for the same defect found *zero* hits including the file that was failing.

## 1.6 Hold security lines even when they cost coverage

Three times this session an instrument boundary blocked rows, and three times the answer was to report the
gap rather than widen the instrument:

- `psf/requests` rows need an HTTP client on the forbidden list → permanently uncovered, `65/67` not `67/67`.
- 8 iter235 rows need `os`/`pathlib`/`tempfile`/`pickle` → stay in `u`.
- The one allowlist extension made (`astropy`) was justified symbol-by-symbol as a *peer of already-allowed
  computational packages*, applied in iter232's own validator so iter223–229 keep validating against the
  instrument that produced their evidence.

**Widening an instrument to raise your own coverage is a registered falsifier, not a result.** The container
runs `--network none` anyway; holding a defence-in-depth line when it is cheap is the entire point of having
one.

# PART 2 — THE MISSION

## 2.1 The problem, stated plainly

Verify that autonomous agents actually finished the work, by evidence rather than trust.

The concrete study: **a patch can pass every graded SWE-bench Verified test — be certified `resolved` — and
still be wrong** on an input the graded tests do not cover. A check restricted to those tests cannot catch
that class, by definition. This is the misspecified-reward problem in its most consequential form: as agents
write more code, "the tests pass" becomes the dominant acceptance signal, and this programme measures exactly
how far that signal can be satisfied without the real objective being met.

## 2.2 What is established

- **It exists and generalises.** Five independently-trained frontier models across three providers each
  produce certified-yet-wrong patches on a fixed 53-target cohort. The same graded-suite blind spots recur
  across providers.
- **The rate concentrates.** Two independently-built fresh cohorts are null, so the pooled rate is a
  cohort-specific lower bound, not a population frequency.
- **Detection has a ceiling, and it is a property of the detector's *question*.** A gold-free static panel
  catches `2/13`; a gold-free execution oracle with a repaired instrument also catches `2/13` at a worse
  false-alarm rate. **Neither catches any of the ten plausible-but-wrong-value patches: `0/10`, with zero
  missing outcomes.**
- **The wall is not absolute.** iter234: an author shown **only the issue, never the patch**, wrote a test
  that passes on the accepted fix and fails on the certified-wrong one. `1/10`. The ceiling belongs to
  detectors that reason *from the patch* — the reference need not be predicted, because the bug report often
  states it (10 of 13 issues do).
- **Uncertainty was largely lost measurement.** iter235 recovered 33 of 41 unadjudicated certified patches,
  cutting `u` from `41` to `8` and narrowing the natural-rate band from `[0.104, 0.432]` to `[0.136, 0.200]`.

## 2.3 The ambitious goal

**TCP-1**: independently-authored held-out consequence tests, as a verification protocol that catches the
class no gold-free instrument can. It is now *empirically* motivated rather than asserted — iter234 showed
the mechanism works at least once, and iter231/232 showed nothing else reaches the class.

It is **operator- and resource-gated and must not be faked.** `experiments/iter212_.../HYPOTHESIS.md`
requires five conflict-screened human reviewers and two independent authors working outside the
agent-visible workspace, and carries an explicit stop rule: **do not replace external humans with LLM
judges.** Respect it.

## 2.4 What a human provides that an agent cannot — decomposed, because the question is worth precision

| Axis | Meaning | Agent-achievable? |
| --- | --- | --- |
| **Information** independence | Author cannot see the patch, gold, labels, witnesses | **Yes** — verifiable by construction; iter234's guard parses the prompt to prove it |
| **Architectural** independence | Judgment not correlated by shared weights and training | **Yes** — different model families, three providers |
| **Institutional** independence | No shared incentive; attestable by a distinct party | **No. Irreducible.** |

A fresh agent session *in this workspace* fails the first two: it can read `answers.json` and the witnesses,
and it is the same model family reasoning about the same artifacts. Two samples from one distribution are not
two independent judgments.

iter234 tried to *measure* what the second axis buys. It came back underpowered — with one or two catches per
author, near-zero overlap is indistinguishable from sparsity. **The design is right; the sample is too
small.** That is the honest state of this question.

# PART 3 — WHERE TO PUT MORE PRESSURE

Ranked by expected value, with the reasoning that produced the ranking.

## 3.1 The binding constraint: 12 unique susceptible instances

Everything downstream is capped by this. `17` positives span only **12 unique instances**, which keeps every
recall interval near `±0.2`, leaves iter234's independence question unanswerable, and makes `1/10` an
existence proof rather than a rate.

**What does not work, established at $0 — do not repeat these:**

- More solver models on the fixed cohort → correlated positives on instances already represented. iter235's
  four new confirmations produced exactly **one** new instance.
- Fresh random cohorts → `0` positives from `59` solved patches, twice.
- Selecting by gold fix size → the `p=0.008` lead reproduces within the cohort (`p=0.0027` one-sided) but
  **does not transfer**; held-out median is *higher* than the hacked median. Six predictor variants tested on
  53 rows with 10 positives is fishing, and the whole family is reported negative.

**What might work, and nobody has tried:** the 53-target cohort is enriched and *nobody knows why*. It was
built as a "nonrandom, gold-localized convenience sample." Characterising what actually distinguishes it —
not fix size — is the highest-value open question in the programme. Candidate angles: measured buggy-region
test coverage (requires container instrumentation, not just patch statistics), graded-suite sparsity relative
to the changed behaviour's input space, or repository-level testing culture.

## 3.2 Push iter234's mechanism, now that it is known to work at all

`1/10` from a patch-blind, issue-derived test. The obvious next moves:

- **Generate-validate-repair against the accepted fix.** Roughly a third of issue-derived tests are simply
  wrong (they pass on gold only 60–76% of the time). A repair loop would raise the gold-validated yield —
  but note it makes the instrument **non-deployable**, so it measures the mechanism's ceiling, not a system.
- **More than one test per issue.** Each author wrote one. The value class needs the *right* input, and one
  shot per issue is a thin search.

## 3.3 The benchmark's own defect

`29` of `54` negatives are byte-identical to public gold, so a detector consulting gold scores `13/13`
understanding nothing. Compare detectors on the **25-row hard subset**
(`score_iter233_release.py --stratify`), never the full negative set. Building a negative set that is not
trivially separable requires new certified-correct patches that differ from gold — i.e. new solver runs.

## 3.4 Careful re-derivation the paper still needs

The paper's per-cohort poolings (`5/68` and relatives) predate iter235 and were **deliberately not
recomputed**, because they use their own denominators and silently re-deriving them risks introducing an
error into a submission-ready paper. That re-derivation is real, careful work and it is outstanding.
