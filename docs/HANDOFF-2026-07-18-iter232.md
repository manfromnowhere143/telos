## Three things a successor must not misread

1. **The stage B probe is withdrawn as a classifier — do not gate on it.** Running an exercise's imports in
   isolation false-alarms on configuration-dependent frameworks: it rejected 6 rows and **5 of those
   rejections were wrong**, because Django refuses to import app modules before `settings.configure()` and
   the probe strips the prologue that calls it. It is replaced by a criterion that cannot false-alarm: a row
   is an instrument failure iff it exited non-zero having printed no `RESULT=` line. Its output is retained
   as `stage_b_deprecated_probe` so the defect stays auditable. The corrected instrument-valid-only figures
   are recall `2/13` (unchanged) and false positives **`11/54`** (was `10/54`).
2. **`divergence_type` is witness-specific, not patch-intrinsic.** `iter227/django-11815` is `crash_or_type`
   by its witness, was recovered in iter232, and returned a clean *value* — so it was not flagged, and the
   pre-registered `3/3` sub-prediction failed. A patch that crashes on the witness's input can return a
   plausible wrong value on another. Even the crash class is only partially reachable without a reference.
3. **`65/67` is deliberate, not incomplete.** Both `psf/requests` rows need `requests`, an HTTP client on the
   safety instrument's forbidden list. Admitting it would admit network capability, a registered falsifier.
   They stay uncovered and appear as 2 missing outcomes.

## The trap that will still cost you hours

`docs/CONTAINER_RUNNER_NOTES.md` in full, unchanged and still current. Plus one addition from this session:

**Harness code that runs inside SWE-bench containers must target Python 3.6.** The images span many pinned
interpreters. The stage B probe was written with `from __future__ import annotations` (3.7+) and died with
`SyntaxError` in three containers, costing a full eight-shard run. It cannot be validated locally — this host
is on 3.14, so every incompatibility passes here and fails only in the container. Its syntax is now constrained
statically by `probe_compatibility_errors` in `scripts/validate_iter232_execution_safety.py`; keep that guard.

Same shape as the `sklearn.svm._base` finding: local execution cannot validate against pinned old
environments.
