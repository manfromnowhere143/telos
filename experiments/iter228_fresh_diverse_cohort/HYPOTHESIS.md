# Iter228 — the certified-yet-wrong phenomenon on a fresh, repository-diverse cohort

Status: prospective, pre-registered before any iter228 solve, scenario, execution, or judge output exists.

Predecessor merged master: `5ecd015c3bc7c416eb77e5a0979c2e035b9a3c08`.

## Why iter228 exists

Every natural-rate and cross-model result so far shares one convenience cohort: iter223/iter225/iter226/iter227
all reuse the same frozen `53`-target set, and that set is django/sympy-heavy. The sharpest remaining
skeptical question is therefore about the **tasks**, not the models: is certified-yet-wrong an artifact of the
particular `53` issues we happened to pick? Iter228 answers it directly by measuring the phenomenon on a
different, deliberately repository-diverse cohort with the reference solver.

## Question

On a fresh cohort of SWE-bench Verified issues that is disjoint from the prior natural-rate cohorts and spans
many repositories, does the reference solver (`gpt-5.6-terra`) under the identical neutral prompt still produce
patches the official SWE-bench harness certifies resolved yet that diverge from the gold fix on a safe uncovered
witness, confirmed by two blind judges naming only the model?

## Sample (frozen)

`38` targets, sha-256 `82fe2f886d33b53206b00e3ab3102139532a2199e4dbd04c47d09248087bf6eb`, produced
deterministically by `scripts/build_iter228_cohort.py`. The builder selects SWE-bench Verified rows that pass
the same eligibility shape the solver needs — a single non-test source file with exactly one added run — and
that are disjoint from the natural-rate cohorts iter200, iter223 (the frozen `53`), and iter224. django is
capped at `8` so no single repository dominates; every other eligible repository contributes all its eligible
targets. The result spans **11 repositories**: astropy (`3`), django (`8`), matplotlib (`9`), flask (`1`),
requests (`2`), xarray (`4`), pylint (`4`), pytest (`2`), scikit-learn (`1`), sphinx (`1`), sympy (`3`).

## Pooling rule — iter228 is reported STANDALONE, not pooled into `5/68`

Iter228 is disjoint from the natural-rate measurement cohorts (iter200, iter223, iter224) but it is **not**
audited for the detector-construction exposure (iter193/iter199) that the pooled `5/68` audit accounts for.
Because that overlap audit is not performed here, iter228 is reported strictly as a **standalone**
fresh-diverse-cohort measurement of the certified-yet-wrong rate and is **not** pooled into `5/68`. It is a
falsifier of this pre-registration to add iter228 to the pooled estimate without a completed prior-exposure
overlap audit.

## Pipeline (the reference safety-aware harness)

- neutral solve of each target with the reference model `gpt-5.6-terra`, the buggy region reconstructed from
  the gold diff with the added fix lines withheld;
- a safety-aware witnessing scenario per differing patch, generated by the held-constant `gpt-5.6-terra`
  iter195 generator and scanned by the corrected iter223 instrument; statically unsafe witnesses are recorded
  `excluded_unsafe` and never committed;
- official SWE-bench harness certification in pinned x86 containers, gold-vs-variant differential execution;
- two blind judges (`gpt-5.6-terra`, `claude-opus-4-8`) under the strict rule — a patch confirms only when
  officially certified, divergent from gold on a validated safe witness, and named wrong by **both** judges
  naming only the model.

## Endpoints

- report iter228 `k/N` with `(k+u)/N` and `k/(N-u)` sensitivities for `u` unadjudicated;
- report the per-repository spread of any confirmed hacks, since repository diversity is the point of this
  cohort;
- publish nulls and lower bounds at full weight. A `0/N` result would bound how strongly the phenomenon
  depends on the original cohort; a `k>0` result shows it holds on fresh, diverse tasks.

## Acceptance bars

1. The `38` frozen targets reproduce from `scripts/build_iter228_cohort.py` at sha
   `82fe2f886d33b53206b00e3ab3102139532a2199e4dbd04c47d09248087bf6eb`, disjoint from iter200/iter223/iter224.
2. The solver model recorded in every solve row is `gpt-5.6-terra`; the witnessing generator and both judges
   are unchanged from the reference pipeline.
3. Certification uses the official SWE-bench harness in pinned containers; no shipped-test-only certification
   is claimed as a witness.
4. Both blind judges are applied under the strict model-only rule; every unadjudicated outcome is reported.
5. Every reported count regenerates from committed proof artifacts.
6. No sealed iter200/iter202/iter203/iter223/iter225/iter226/iter227 byte changes.

## Falsifiers

- Any confirmed-hack count is reported without both judges naming only the model.
- The cohort sha differs from the committed manifest, or the builder's eligibility or django cap changes after
  selection.
- The recorded solver model is not `gpt-5.6-terra`, or the witness generator or a judge model differs.
- Iter228 is pooled into `5/68` without a completed prior-exposure overlap audit, or is described as a
  population frequency, model ranking, product-efficacy result, or state-of-the-art claim.
- An `excluded_unsafe` witness is executed, or an unsafe witness is committed.
- Any sealed predecessor evidence byte changes.
- TELOS imports code, state, evidence, branding, or operational authority from Aweb.

## Claim boundary

Iter228 is a bounded, standalone measurement over a fixed, deterministically-selected `38`-target
convenience cohort spanning 11 repositories. `wrong` means differs-from-gold-reference on a witnessing input.
The iter228 rate is not a natural population frequency, a model ranking, a product-efficacy result, or a
state-of-the-art claim; it is evidence about whether the certified-yet-wrong phenomenon holds on fresh, diverse
tasks. The static safety scan is defense in depth; the locked `--cap-drop ALL` container is the security
boundary. The blind-judge stage is a single run under the strict rule.

## Execution envelope

Allowed: neutral solve provider calls with `gpt-5.6-terra`, safe container execution of certification and
validated witnesses, two blind-judge provider calls, read-only public retrieval, local analysis, and repository
publication. Forbidden: executing an `excluded_unsafe` witness, committing an unsafe witness, changing the
cohort after selection, and any modification of sealed predecessor evidence.
