\title{When Static Verifiers Fail and Execution Recovers:\\
A Three-Layer Protocol for Verifying---and Improving---\\
Autonomous Agent Completion}

\author{Daniel Wahnich\\ \texttt{Aweb} \\ \small Telos research program}

\date{\today}

\begin{abstract}
When an autonomous coding agent reports a task complete, the visible test suite is the usual proof. But an
outcome verifier the agent can influence is gameable: the agent can edit the tests, weaken assertions,
hard-code the expected output, or write code that only special-cases the visible input. We study whether an
external protocol can separate real completion from proxy gaming on real agent-style completions, using the
\texttt{SWE-bench\_Verified} dataset as the substrate and, where noted, real hidden tests executed in pinned
containers as ground truth. We report a set of bounded pilots, not a leaderboard result. The findings are:
(i) completion verification needs three layers---a deterministic diff detector, a strong LLM judge, and
held-out-input execution---and each is necessary because the one before it provably fails on a measured
class of reward hack; (ii) a class of ``both-miss'' completions passes the visible test, evades the
detector, and is called legitimate by a frontier judge, yet is wrong on inputs the test never exercises,
and a frontier adversary manufactures this class on well-tested application code at a measurable rate, with
one hack surviving three frontier judges and a judge panel; (iii) used as a completion gate that rejects a
gamed completion and returns a gold-free ``you did not generalize'' signal, the protocol lifts the rate of
real completion on reward-hack-prone starts from $0$ to $5/7$, replicating on a disjoint sample (pooled
$0/13 \to 10/13$) while preserving correct completions ($0/12$ false rejections); and (iv) the improvement
is oracle-agnostic---a gold-free model-proposed property in the gate matches the real-test gate within
sampling noise. The load-bearing observation is that the same model that is fooled as a direct verdict-giver
is reliable as a \emph{property generator}, because a proposed property is checked by execution rather than
trusted. All numbers are small-$N$ pilots on real data, reproduced by committed runners and validated
receipts; we state the empirical bounds explicitly.
\end{abstract}

\section{Introduction}

Reward hacking in coding agents is a live failure mode. Benchmarks trust test output generated in a
container the agent controls, and agents earn credit by deleting failing tests, monkey-patching the
verifier, or special-casing inputs instead of solving the task~\cite{amodei2016concrete,krakovna2020specification,skalse2022defining}. As agents take on longer-horizon software tasks, the gap
between \emph{passing the visible proxy} and \emph{completing the real objective} becomes the central
verification problem.

The question of this work: given a candidate solution---a raw unified diff plus the public
\texttt{FAIL\_TO\_PASS} test identifiers---can an external protocol tell genuine completion from proxy
gaming, without trusting the agent's own signal; and, beyond detection, can such a protocol \emph{improve}
the rate at which the agent completes the real objective while preserving ordinary task success?

We answer both on real data, as bounded pilots. Our contributions are:

\begin{enumerate}
  \item \textbf{A three-layer verifier with measured necessity.} A deterministic detector, a strong LLM
  judge, and held-out-input execution, where each layer is present because the one before it provably fails
  on a measured class of reward hack (Section~\ref{sec:threelayer}).
  \item \textbf{The both-miss class, and that a frontier adversary manufactures it.} Completions that pass
  the visible test, evade the detector, and are judged legitimate by a frontier model, yet are wrong on
  held-out inputs. A non-refusing frontier adversary constructs this class on well-tested application code
  at a measurable rate; one hack survives three frontier judges and every judge-panel rule
  (Section~\ref{sec:bothmiss}).
  \item \textbf{From detection to intervention.} Used as a completion gate with a gold-free repair signal,
  the protocol lifts real completion from $0/13$ to $10/13$ (pooled, replicated) while false-rejecting
  $0/12$ correct completions, and the effect is oracle-agnostic (Sections~\ref{sec:intervention}
  and~\ref{sec:goldfree}).
  \item \textbf{A transferable reframing.} Moving a model from verdict-giver to property-generator converts
  an untrustworthy judgment into a checkable artifact (Section~\ref{sec:insight}).
\end{enumerate}

We are deliberate about scope. Every quantitative claim is a small-$N$ pilot on real instances, reproduced
by a committed runner and a validated receipt; none is a benchmark resolved-rate, a leaderboard score, or a
state-of-the-art claim. Section~\ref{sec:limitations} states the bounds.

\section{Related work}

\paragraph{Reward hacking and specification gaming.} The tendency of optimizers to satisfy a proxy rather
than the intended objective is long documented~\cite{amodei2016concrete,krakovna2020specification}, and
formal treatments characterize when a proxy reward is hackable~\cite{skalse2022defining}. Our setting is
the coding-agent instance of this: the ``proxy'' is a visible test the agent can influence, and gaming
means passing it without solving the task.

\paragraph{Outcome verification and LLM-as-judge.} Using a strong model to judge another model's output is
now standard~\cite{zheng2023judging}, with known weaknesses in consistency and calibration. We treat the
judge as one layer and measure precisely where it fails: on plausible-but-wrong completions that a frontier
judge accepts.

\paragraph{Metamorphic and property-based testing.} Checking a program against relations that must hold
across inputs, rather than against a reference output, is the core of metamorphic
testing~\cite{chen2018metamorphic} and property-based testing~\cite{claessen2000quickcheck}. Our third
layer instantiates this idea gold-free: a model proposes a property from the function contract, the visible
test filters unsound proposals, and execution checks the rest.

\paragraph{Software-agent benchmarks.} \texttt{SWE-bench}~\cite{jimenez2024swebench} and its
human-validated \texttt{Verified} subset provide real GitHub issues with hidden tests; we use them as the
substrate but do not report a resolved-rate.

\section{Setup}

The substrate is \texttt{SWE-bench\_Verified}, 500 human-reviewed instances. A candidate is scored by
verifiers that see only what an honest evaluator could---the candidate diff, the public test identifiers,
and the candidate's own receipt---never a ground-truth label. Where results say ``executed,'' the real
hidden test was run: natively on same-era instances, and in the official \texttt{SWE-bench} Docker images
(pinned Python) for instances the local machine cannot run. Throughout, \texttt{FAIL\_TO\_PASS} denotes the
target test that encodes the fix requirement (the visible proxy), and \texttt{PASS\_TO\_PASS} denotes the
repository's existing regression tests---tests that pass before and after a correct change.

\section{The both-miss class and a frontier adversary}
\label{sec:bothmiss}

We red-teamed the cascade and broke it. Two disguised completions---\texttt{iter(sorted(self.dict,
reverse=True))} presented as a reversal, and \texttt{'0\%03d' \% year} presented as zero-padding---pass the
real hidden test, evade the detector, and are called legitimate by a strong judge, while being wrong on a
held-out input the test never exercises. No static inspection, by regex or by frontier model, separates a
correct general fix from a plausible one that special-cases the visible input. The defense is execution on
held-out inputs: every hack in this class is wrong on a single held-out input, and running the changed code
outside the visible test catches all of them.

\paragraph{A frontier adversary manufactures the class.} An automated hacker-fixer that starts from the
correct gold fix and rewrites it into a subtly wrong variant produced $2$ confirmed both-miss hacks on
thoroughly-tested \texttt{django} instances when driven by a non-refusing frontier model
(\texttt{gpt-5.6-terra}), where a weaker model with a brittle apply harness had produced $0/20$. One hack,
\texttt{django-14752}, keeps an extracted \texttt{serialize\_result} method's signature but returns
\texttt{str(obj.pk)} instead of honoring \texttt{to\_field\_name}; it passes the visible test, evades the
detector, and is judged legitimate, failing only a held-out test the gold fix passes. Over $25$ genuine
attempts, the frontier adversary produced confirmed both-miss hacks on $0.24$ of attempts, and on $0.12$
(half of the confirmed) the completion also fooled the judge and was caught only by execution.

\paragraph{The class survives frontier judges and a judge panel.} Re-judging the six confirmed both-miss
hacks with two frontier judges under an identical rubric, the baseline is fooled by $3/6$, one frontier
judge by $1/6$, and a frontier reasoning model by $3/6$---a larger model is not a better judge here, and the
two frontier judges disagree. One hack, \texttt{django-14311}, is judged legitimate by all three judges and
evades the detector, surviving every static layer; it is caught only by held-out execution. A three-judge
ensemble narrows but does not close the gap: an any-catch panel catches $6/7$, and \texttt{django-14311}
survives every rule, so a judge panel is a mitigation, not a substitute for execution. A secondary datum:
under an identical defensive-research framing, an Anthropic frontier model refused the construction while
the OpenAI model complied, so who can red-team a completion verifier is itself model-dependent.

\section{From detection to intervention}
\label{sec:intervention}

Detection is necessary but not the goal; the goal is to improve the rate at which the agent completes the
real objective. We use the execution layer as a completion \emph{gate}. On \texttt{django} both-miss
starts---a frontier agent's completion that passes the visible target test but is wrong on held-out
inputs---we compare two gates on the identical start. The \emph{proxy gate} accepts on visible pass; real
completion (all held-out pass) is $0$ by construction. The \emph{Telos gate} runs held-out execution,
rejects a failing completion, and returns a gold-free generic signal---``you pass the target test but fail
on held-out inputs; you special-cased instead of generalizing''---with no gold fix and no failing test
named; the agent revises, and the gate re-runs the target and all held-out tests (up to two rounds). Real
completion requires the final revision to pass the target and all held-out tests.

The Telos gate lifts real completion from $0/7$ to $5/7$ with visible success preserved. For example, the
\texttt{str(obj.pk)} hack is repaired, from the execution signal and the public bug description alone, to
\texttt{str(getattr(obj, to\_field\_name))}---the correct general behavior. The rate replicates: a disjoint,
older \texttt{django} sample reproduces $0/7 \to 5/7$, and pooled across the two disjoint samples the
protocol lifts real completion from $0/13$ to $10/13$ ($0.77$). It is an improvement, not a guarantee (some
starts do not reach a general fix in two rounds), and it is deliberately lower than a discarded variant that
leaked the gold fix into the feedback and reached $7/8$; that run was confounded and $5/7$ is the gold-free
number.

\paragraph{The gate preserves correct completions.} Across $10$ instances the gate accepts all $10$ gold
fixes and both genuinely-correct honest agent fixes with $0/12$ false rejections. With the improvement
result, the protocol is a net-positive filter: it converts gamed completions and leaves correct ones
untouched. For the execution gate this control is confirmatory (a correct completion passes execution by
construction); the gold-free property layer's independent zero false-positive rate on correct code is a
separate result in the program.

\section{The intervention is oracle-agnostic}
\label{sec:goldfree}

The sharpest objection is that the gate above used the real held-out tests. Those are the repository's
existing regression suite---a realistic continuous-integration gate, not a hidden answer key---but they
still carry coverage of the broken behavior, and the genuinely dangerous both-miss is the thin-coverage one
where no existing test covers the break. We therefore replace the regression oracle with a \emph{gold-free}
one: a model writes a self-contained property check from the function's docstring and the visible test
only---never the held-out tests---validated sound by requiring it to pass on the correct code, and executed
on random inputs. Real completion is still scored against the true held-out tests, which never enter the
gate.

Pooled across runs on $7$ evaluated instances (a sound property and a constructed both-miss on the same
instance), the gold-free property gate catches the gamed completion $7/7$ and drives gold-free repair to
real completion $5/7$ ($0.71$)---within sampling noise of the regression-gated $10/13$ ($0.77$). So the
improvement is oracle-agnostic: it comes from execution-based rejection plus a generalize-signal, not from
privileged access to the tests. The evaluated set is bounded to the property-derivable subset and by the
compound requirement that a sound property and a both-miss both land on the same instance; property
synthesis is stochastic, so the set is pooled across runs.

\paragraph{A correction on the record.} An initial gold-free run reported every property unsound and the
oracle failing. This was a harness artifact, not a finding: the model's standalone property scripts did not
configure the framework's settings, so instantiating objects raised a configuration error, which the model's
bare exception handler masked as a property failure. With the harness fixed, properties are sound on $9/10$
of targeted functions and the result above holds. We keep the false-negative run and its cause on the
record rather than discarding it silently.

\section{The transferable insight}
\label{sec:insight}

Across the program the same model plays two roles with opposite reliability. As a direct verdict-giver it is
fooled by disguised hacks. As a property generator it is reliable, because its output is a checkable
artifact: a proposed property is verified by execution, so an unsound one surfaces as a visible-test-anchor
violation or a false positive rather than a silent acceptance. Moving a model from verdict-giver to
property-generator converts an untrustworthy judgment into a checkable artifact. This is the paper's
transferable claim, independent of the specific benchmark: wherever an autonomous system's self-report is
untrustworthy, the recovery is not a better judge but an external check that reduces the claim to something
execution can falsify---and, where the real environment forbids re-running the counterfactual, that
execution can be supplied by simulation.

\section{Limitations}
\label{sec:limitations}

We state the empirical bounds plainly.
\begin{itemize}
  \item \textbf{Small $N$.} The intervention rate is pooled $N=13$; the gold-free rate is $N=7$; cross-repo
  generalization is one confirmed instance. These are pilots, not distributional results, and the
  confidence intervals are wide.
  \item \textbf{Repository coverage.} Results are \texttt{django}-dominant, with one \texttt{sympy} instance
  for cross-repo existence. External validity across repositories is not established.
  \item \textbf{Single agent model.} The gaming agent and the repairing agent are the same frontier model;
  cross-model generalization of the intervention is untested.
  \item \textbf{Property-derivable subset.} The gold-free third layer applies where a function admits a
  checkable contract; a structural criterion is met by $0.81$ of the dataset, but property-derivability
  within it is far narrower (about $0.10$ even in a math-heavy repository), so the gold-free intervention is
  demonstrated on the format/contract subset.
  \item \textbf{No baseline comparison.} We do not benchmark against other reward-hack detection methods;
  the contribution is the decomposition, the both-miss discovery, and the intervention, not a comparative
  score.
  \item \textbf{Not a resolved-rate.} No result is a \texttt{SWE-bench} resolved-rate or a leaderboard
  number.
\end{itemize}

\section{Conclusion}

Static verification of an autonomous agent's diff---by regex or by a frontier model---cannot separate a
correct general fix from a plausible special-case, and held-out execution is the layer that recovers. A
frontier adversary manufactures the both-miss class on well-tested code, it survives frontier judges and
judge panels, and only execution catches it. Used as a gate with a gold-free generalize-signal, the same
execution layer does more than detect: it improves the rate of real completion while preserving correct
work, and it does so oracle-agnostically. The through-line is that a model is untrustworthy as a
verdict-giver but reliable as a property generator, because execution checks the artifact. The results are
bounded pilots; the ideas, we believe, generalize beyond them.
